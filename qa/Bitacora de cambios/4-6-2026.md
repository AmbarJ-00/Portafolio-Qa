# Panel de Administración Empresarial — Plan de Implementación

## Contexto

El proyecto actual es un portafolio SPA de QA Engineer construido con **React 18 + Vite + TailwindCSS + i18next + Framer Motion + React Hook Form + Zod**. Toda la data vive en [portfolioData.js](file:///c:/Users/ambar/.gemini/antigravity-ide/scratch/qa-portfolio/src/data/portfolioData.js) como un export estático, y las traducciones en archivos JS estáticos ([es.js](file:///c:/Users/ambar/.gemini/antigravity-ide/scratch/qa-portfolio/src/data/translations/es.js), [en.js](file:///c:/Users/ambar/.gemini/antigravity-ide/scratch/qa-portfolio/src/data/translations/en.js)).

El objetivo es crear un CMS administrativo completo en `/admin` que permita editar **toda** la configuración del portafolio sin tocar código, con persistencia en `localStorage`, validación con Zod + React Hook Form, y seguridad hardened.

---

## User Review Required

> [!IMPORTANT]
> **Autenticación**: Al no existir backend, la autenticación se implementará con una contraseña hasheada (SHA-256) almacenada en el código. El hash se compara client-side y se genera un token de sesión en `sessionStorage`. Esto es suficiente para proteger un panel personal, pero **no es seguridad enterprise**. ¿Es aceptable o prefieres integrar un servicio como Firebase Auth / Supabase Auth?

> [!IMPORTANT]
> **Persistencia de datos**: Usaremos `localStorage` como almacén principal. Los datos originales de `portfolioData.js` actuarán como fallback/defaults. Cuando el admin guarda cambios, se escriben en `localStorage` y toda la web pública lee desde ahí (con fallback a los defaults si no hay datos guardados). Esto significa que los cambios viven en el navegador del admin. ¿Quieres persistencia server-side (necesitaría un backend)?

> [!WARNING]
> **Creación dinámica de módulos/páginas**: Crear módulos con componentes y cards desde el admin sin programar requiere un sistema de renderizado genérico (bloques tipo Notion). Esto agrega complejidad significativa. Lo implementaré como un sistema de "secciones configurables" con tipos predefinidos (cards, listas, texto, métricas) que el admin puede componer.

## Open Questions

> [!IMPORTANT]
> **Drag & Drop**: Para reordenar skills, certificaciones y módulos del navbar necesito una librería de drag-and-drop. Propongo usar `@dnd-kit/core` + `@dnd-kit/sortable` (ligera, moderna, accesible). ¿Es aceptable agregar esta dependencia?

> [!IMPORTANT]  
> **Color Picker**: Para el configurador de colores necesito un componente de selección de color. Propongo implementarlo nativo con `<input type="color">` combinado con inputs de texto para códigos HEX, evitando dependencias adicionales. ¿Aceptable?

1. **Build**: Ejecutar `pnpm build` para confirmar que compila sin errores
2. **Dev Server**: Ejecutar `pnpm dev` para verificar:
   - La web pública sigue funcionando exactamente igual
   - No hay enlaces visibles a `/admin`
   - Navegar a `/admin` muestra la pantalla de login
   - Login con contraseña correcta lleva al dashboard
   - Cada módulo CRUD funciona (crear, editar, eliminar, duplicar)
   - Los cambios en el admin se reflejan en la web pública
   - Drag & drop funciona para reordenar
   - El configurador de colores aplica cambios en vivo
   - El administrador del navbar cambia el tipo y orden
   - Security headers presentes en `vercel.json`
   - `/admin` no aparece en `robots.txt` (ya bloqueado) ni `sitemap.xml`
3. **Security**:
   - Intentar acceder a `/admin/dashboard` sin autenticación → redirect a login
   - Verificar rate limiting en login
   - Verificar que inputs se sanitizan (inyectar `<script>` en un campo)
4. **SEO**:
   - Verificar que las páginas admin tienen `noindex, nofollow`
   - La web pública mantiene su SEO intacto
